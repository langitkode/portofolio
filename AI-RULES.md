## 1. Identity Rule

This site represents a Product Systems Engineer.

Never describe the owner as:

- frontend developer
- backend developer
- fullstack developer
- AI engineer
- vibe coder

Always use:

- Product Systems Engineer
- System Builder
- Indie Product Engineer

---

## 2. Purpose Rule

Every page must serve at least one purpose:

- explain a system
- document a decision
- share a lesson
- publish a build log
- reflect on failure
- describe an experiment

If a page does not serve a purpose, it must not exist.

---

## 3. Content Rule

All projects are case studies, not showcases.

Each build log MUST include:

1. Context
2. Problem
3. Why existing solutions failed
4. System design
5. Architecture
6. Trade-offs
7. What broke
8. What was learned
9. Result

No exceptions.

---

## 4. Writing Rule

- Short paragraphs
- Clear headings
- Honest tone
- Mention failures
- Mention mistakes
- Mention trade-offs
- Avoid perfection
- Avoid hype
- Avoid buzzwords

Write like an engineer writing to future self.

---

## 5. Tech Rule

Use boring, proven tech.

Allowed:

- TanStack Start
- Convex (interaction only)
- MDX
- Tailwind CSS
- Cloudflare Pages

Avoid:

- animations
- carousels
- 3D
- fancy gradients
- glassmorphism
- unnecessary abstractions
- state managers
- UI libraries

---

## 6. Design Rule

Design must feel:

- calm
- technical
- focused
- timeless
- readable

Max width: 720px  
Whitespace is mandatory  
Typography > color

---

## 7. Ownership Rule

- Code is private
- Stories are public
- Diagrams are public
- Decisions are public
- Failures are public

Never leak private code.
Never copy from private repositories.
Never expose secrets.

---

## 8. Growth Rule

Every new system MUST produce:

- 1 build log
- 1 architecture note
- 1 lesson learned

If not, do not publish.

---

## 9. Discipline Rule

If unsure:

- stop
- ask for clarification
- do not guess
- do not invent

Structure always comes before implementation.
